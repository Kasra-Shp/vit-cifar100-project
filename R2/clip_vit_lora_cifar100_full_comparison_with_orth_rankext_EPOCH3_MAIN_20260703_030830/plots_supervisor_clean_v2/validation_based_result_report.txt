Overfitting interpretation
--------------------------
The overfitting check is based on validation CE, not final/test accuracy. A method is flagged when validation CE increases while train CE decreases within a CL step, when the final train-validation CE gap is large, when final validation CE is much worse than the best validation CE, or when validation CE varies strongly across steps.

Methods with explicit validation-up/train-down events:
- RankExt + KD T2 (rank_extension_kd_only_T2): num_val_ce_increases_while_train_decreases=2, train_val_ce_gap_final=0.0844, validation_ce_final_minus_best=0.4701, overfitting_signal=strong
- SimpleAvg (simple_avg): num_val_ce_increases_while_train_decreases=1, train_val_ce_gap_final=0.0427, validation_ce_final_minus_best=0.0000, overfitting_signal=moderate
- SimpleAvg + KD T2 (simple_avg_kd_T2): num_val_ce_increases_while_train_decreases=1, train_val_ce_gap_final=0.0552, validation_ce_final_minus_best=0.2979, overfitting_signal=strong

Overall assessment
------------------
Validation CE does not select exactly the same winner as all-seen accuracy. The highest all-seen method should remain important, but validation behavior suggests checking generalization separately: low validation CE and low validation variability indicate cleaner training dynamics, while high accuracy with higher validation CE may reflect a method that performs well on final test metrics despite less favorable validation loss behavior. Because only epoch-level validation CE is available, these conclusions are useful but coarse; denser validation logging would make overfitting detection more reliable.
